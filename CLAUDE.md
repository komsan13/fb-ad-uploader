## กฎข้อที่ 1: ห้ามเดา ต้องหาข้อมูลก่อนเสมอ

**ก่อนสรุปสาเหตุของอะไรก็ตาม ต้องมีหลักฐานที่วัดได้ ไม่ใช่เหตุผลที่ฟังดูสมเหตุสมผล**

ระบบนี้ใช้เงินโฆษณาจริงอัตโนมัติ การเดาผิดหนึ่งครั้งแปลว่าเงินหายจริง หรือบัญชีโดนแบนจริง

### ทำแบบนี้

1. **หาเอกสารทางการก่อน** — Meta Marketing API, Anthropic API ใช้ `claude-api` skill
   อย่าตอบจากความจำเรื่อง API ของคนอื่น
2. **วัดของจริง** ไม่ใช่อ่านโค้ดแล้วสรุป
   - หน้าเว็บ/พิกเซล/JS → `gstack browse` โหลดหน้าจริงแล้วดู network + console
   - เส้นทาง FB API → เรียก API จริงด้วย token จริง (อ่านอย่างเดียว ปลอดภัย)
   - สถานะบนเซิร์ฟเวอร์ → ssh เข้าไปดู ไม่ใช่เดาจากโค้ด
3. **แยกให้ออกระหว่าง "แก้ปัญหาได้" กับ "เจอสาเหตุแล้ว"**
   บางทีแก้ถูกโดยบังเอิญแต่สาเหตุเป็นอีกเรื่อง ถ้ายังไม่พิสูจน์ ต้องบอกว่ายังไม่รู้
4. **บอกตรงๆ เมื่อยังพิสูจน์ไม่ได้** ดีกว่าเสนอทฤษฎีที่ฟังดูดี

### เคสจริงที่พลาดมาแล้ว (19 ก.ค. 2026)

| เดาว่า | ความจริง |
|---|---|
| Meta หาพิกเซลไม่เจอเพราะ `/lp` อยู่หลัง basic auth | basic auth เป็นปัญหาจริงและร้ายแรง (คนกดแอดเจอหน้าใส่รหัส) **แต่ไม่ใช่สาเหตุของ error นั้น** |
| ใช้ `last_fired_time` ของพิกเซลยืนยันว่า event เข้าไหม | ค่านี้ **cache และอัปเดตไม่สม่ำเสมอ** ให้ผลขัดแย้งกันเอง เสียเวลาไล่เช็คสองรอบ |

ตัวที่ให้คำตอบจริงคือ **Events Manager → Test Events** และการโหลดหน้าด้วยเบราว์เซอร์จริงแล้วดู network

ระหว่างไล่หาสาเหตุด้วยการวัดจริง ถึงเจอบั๊กที่มีอยู่จริง: `fbq('track', ...)` แบบไม่ระบุพิกเซล
ยิงเข้าทุกพิกเซลที่ init ไว้พร้อมกัน มี 10 พิกเซล + 10 บล็อก = PageView ถูกนับเกินจริงหลายเท่า
ถ้าไม่เจอ ตัวขยายงบกับตัวปิดแอดขาดทุนจะตัดสินใจจากตัวเลขที่เฟ้อ

## กฎข้อที่ 2: แตะของ production ต้องระวัง

- **ห้ามใส่คอมเมนต์กลางคำสั่งที่ต่อบรรทัดด้วย `\`** (เช่น `docker run`) คำสั่งจะขาดกลาง
  เคยทำเว็บล่มและรหัสผ่าน basic auth หายมาแล้ว เพราะ `redeploy.sh` ดึง hash จาก container เดิม
- รัน `bash -n <script>` ก่อนเอาสคริปต์ไปรันบนเซิร์ฟเวอร์จริงเสมอ
- ทดสอบด้วย `CONFIG_PATH` ชี้ไป scratchpad **อย่ายิงใส่ไฟล์ config จริง**
- ก่อนแก้ config บนเซิร์ฟเวอร์ ให้สำรองก่อน (`cp config.json config.json.bak-<เหตุผล>`)
